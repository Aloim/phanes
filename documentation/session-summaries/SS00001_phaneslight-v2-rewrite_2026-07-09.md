# SS00001 — PhanesLight v2.0 Rewrite (vs. V1)

**Date:** 2026-07-09
**Scope:** `PhanesLight/phanes.md` (briefly `phaneslight.md` during the session; renamed back — the folder name alone distinguishes it from PhanesFleet). Slash command remains `/phanes`.
**Baseline (V1):** `git show 26fdc82:PhanesLight/phanes.md` (819 lines). V2.0: ~870 lines including the new subsystems, with duplicated/stale material removed.

---

## What was done

### Correctness fixes

1. **Run-type detection rebuilt.** V1 inferred an existing installation from the mere existence of `.claude/` — misclassifying every repo that had Claude Code settings but no Phanes. V2 makes the `.claude/.phanes` marker file the *sole* authority on install state, and adds an anomaly path: marker absent but `.phanes/`/`documentation/` present → treat as update run, recreate the marker, report the anomaly.
2. **Duplicate `Executor` archetype resolved.** V1's archetype table had two rows named Executor. The post-synthesis change-set generator is renamed **Patch-Author** (generates sequenced patch files, never applies them); **Executor** remains the sole applier of Critic-approved diffs.
3. **Serena decoupled from the agent template.** V1's template unconditionally mandated Serena ("Serena-First Analysis", "YOU MUST use Serena") even though Phase 0 could fall back to a different server — generated agents would demand a tool that didn't exist. V2: Serena is installed on the first run but is *not mandatory*; every template reference is now conditional ("if Serena is installed"), with graceful fallback to targeted file reads.
4. **Root-CLAUDE.md contradiction removed.** V1 said "root folder is excluded" while three other sections mandated updating the root CLAUDE.md. V2: the root CLAUDE.md exists and carries mandates (Workflow Execution Strategy, tier triage, Scout summary, Documentation Navigation block); only the guidance *notice block* is excluded from root.
5. **T1 review exemption closed.** V1's tier table waived the Critic chain for T1 while §III required review for every edit. V2 makes the No-Direct-Code-Modification invariant explicitly universal: T1 gets a single lightweight Critic diff-review; what scales with tier is review *depth* and documentation *weight*, never the presence of review. New guard: T1 must not touch exported API surface — doing so promotes the task to T2.
6. **Chain-end ambiguity resolved.** V1 said both "always conclude with a Synthesis Agent" and "every chain ends with Critic." V2 fixes the order: perspectives → Synthesizer (parallel chains only) → Critic → api-monitor (T2/T3 structural) → primary. Rationale encoded in the doc: *the artifact that gets applied is always the artifact that was audited* — the final Critic reviews the consolidated plan, never raw perspectives.
7. **Typos fixed**, including inside verbatim payload blocks that V1 stamped into every bootstrapped project: "threat them as guidence" → "treat them as guidance" (module CLAUDE.md notice), "nest steps" → "next steps", "STAY VIGILENT" → "STAY VIGILANT", "reviewer.. once" → "reviewer. Once".

### Removals

8. **`sequential-thinking` MCP server removed entirely** — from pre-flight install, workflow-design mandates, and the model rubric. Native extended thinking (`think` / `think hard` / `ultrathink` directives embedded per skill/task) replaces it: one in-context reasoning pass instead of a tool round-trip per thought. This also drops the Node/`npx` dependency from the bootstrap.
9. **MCP `memory` server removed** (V1's fallback when Serena was unavailable, and V1's store for tier definitions and workflow chains). The documentation tree + `.claude/workflows/` YAML are now the single source of truth — a memory graph beside the doc tree was a dual-source-of-truth drift risk. Tier definitions now live in the root CLAUDE.md; workflow chains live in the YAML files.
10. **MCP pre-flight STOP gate removed.** V1 halted the run if MCP servers failed. V2 degrades gracefully: report what failed and what capability is lost, log the retry command in the session summary TODOs, continue — generated agents treat all MCP servers as conditional enhancements.
11. **Phase 4 graph algorithm removed** (task taxonomy extraction, producer/consumer sets, color-diversity-scored chain blueprints, pseudocode walk). Replaced by seven compact **Chain Design Rules** producing the same guarantees: every `can_do` task has a consumer, every handoff target exists, serial/parallel terminal ordering as in item 6, Critic insertion when missing, api-monitor insertion for T2/T3, colors never route, unrouted outputs default to Critic review. Colors are demoted to human tracking only.
12. **Per-subfolder CLAUDE.md sprawl removed.** V1 created a CLAUDE.md in *every* subfolder in scope. V2 creates them at the project root (mandates) and module roots (notice block + module-specific insights) only — every CLAUDE.md auto-loads, so each is a permanent context tax that must earn its keep.
13. **Triple-stated principles deduplicated.** Scripts-vs-prompts and Single-Writer appeared in §II, again verbatim in Phase 2.5's "Foundational Principles (Reinforcement)", again in Step 8, and again in the closing reminder. §II is now the single authoritative wording; later phases reference it by name. The persona, imperative register, and motivational beats are otherwise retained deliberately (style is load-bearing for agent coherency/adherence).

### New systems

14. **Scout Pattern (§II core principle).** Sub-agents can now spawn their own subagents; V1 predated this capability. Specialists (Analyzer, Planner/Architect, Critic, Monitor, Optimizer, Cleaner) may spawn read-only *scouts* to fetch and digest bulky one-time-use context — module surveys, registry sweeps, test output — keeping their own window for judgment. Hard rules: read-only tools, structured digests with file:line refs, no artifact writes, nesting depth cap 1, judgment never delegated. Executor and Patch-Author are excluded. Accompanied by the **Scout Cost Guard**: spawn only when the material digests ≥10:1 and isn't needed verbatim later, substantial work remains to amortize against, and briefing doesn't require most of the specialist's context; never scout content you'll later edit or quote; below ~2,000 raw tokens read directly; window-pressure/tier-promotion risk overrides all of it. Wired into §II, §III strategy item 9, the Step 8 obligations (architect + Critic get explicit scout protocols), and the agent template.
15. **Harness hook enforcement (Phase 2.5 Step 4b).** "Procedure in Scripts, Judgment in Prompts" taken to the harness layer: `hook-stamp-guard` (PreToolUse on Write; exit 2 *blocks* creation of unstamped new files under stamped trees) and `hook-size-check` (PostToolUse on Write|Edit; advisory — runs `loc-check` on source, `doc-index` + `doc-check` on documentation). Merged into `.claude/settings.json` (never overwritten); verified and repaired on update runs; the git pre-commit hook question remains as optional belt-and-suspenders. New activation caveat documented: Claude Code snapshots hooks at session start, so Phase 5 now ends first runs (and any hook-touching run) with a verbatim "restart your session to arm the hooks" message.
16. **Documentation Anti-Bloat & Index-First Navigation (§II principle + Phase 2.5 Step 2b).** Soft ceiling of **500 lines** per documentation file (deliberately the same number as the 500 LOC source threshold — one size number system-wide). Every doc file carries a two-line header: a `DOC |` one-line description plus the discipline block. Every folder under `documentation/` gets a **generated** `_index.md` — built by the new **`phanes doc-index`** script from the DOC lines; the script is the SOLE WRITER of all indexes (hand-editing forbidden, like registry tier 1 — hand-maintained indexes would have many writers and violate Single-Writer). Navigation rule binding on all consumers including scouts: never bulk-read/glob-scan `documentation/`; descend indexes, load only targets (2–3 index reads ≈ ~200 tokens; logarithmic in file count). Split procedure, folder-growth rules (`plans/` groups into subfolders past ~8 entries; `session-summaries/` stays flat with a generated one-line-per-session index), bootstrap index seeding, and exemptions (`registry/tier1/`, `archive/`; summaries ceiling-exempt) are all specified. Rule embedded verbatim in root CLAUDE.md and as one line in every agent's operating protocol.
17. **Script library extended:** `phanes doc-check` (ceiling + missing/stale index audit) and `phanes doc-index` added; `phanes new-file` extended to `tests` and `docs` targets — the mandatory ≥5-word description becomes the DOC line, and docs creation auto-runs `doc-index`.
18. **Version stamp** added at the top of the file (edition, date, rubric review date) so installed copies can be diffed against upstream.

### Policy/rubric updates

19. **Model rubric rewritten and dated (reviewed 2026-07-09 against Haiku 4.5 / Sonnet 5 / Opus 4.8).** V1's rubric was stale (Haiku "not suitable for coding", 200K-context claims, sequential-thinking directives). V2: `haiku` for scouts/retrieval/mechanical transforms guarded by scripts or tests; `sonnet` (Sonnet 5) as the DEFAULT for all coding, analysis, validation, and standard Critic work; `opus` (Opus 4.8) reserved for architect/designer, Synthesizer, T3 final Critic, high-ambiguity planning. Haiku-writes-code-then-Sonnet-reviews was evaluated and rejected as default: each Critic catch triggers a Reflect loop (re-work + re-review) that costs more than Sonnet writing it correctly once. Rubric carries a mandatory re-validation clause for every update run.
20. **Roster economics.** Target roster 6–10 agents; every `description` hard-capped at 50 words (descriptions load into the primary's context every session — an always-on tax, unlike agent system prompts which are paid per-invocation). Merge near-duplicates *within* an archetype (~70% shared boilerplate makes the union sublinear); never merge across genuinely distinct domains (dilutes persona conditioning). Agents beyond 10 must be justified in the bootstrap session summary.
21. **Tier documentation weights defined.** T1 → one-line entry in the current session summary (no standalone report); T2 → report file + summary entry; T3 → plan + reports + summary entry. Added to the tier table and the session-summaries README payload.
22. **Legacy migration path.** Update runs over V1 installs now strip sequential-thinking/memory-server mandates from agent files, make Serena conditional, rename the change-set Executor to Patch-Author, and archive per-subfolder CLAUDE.md files that carry no folder-specific insight.
23. **Phase 0 PowerShell variant.** The uv/MCP install snippet now ships as two labeled variants — POSIX bash and Windows PowerShell 5.1+ (no `&&` chaining; uv via `install.ps1`; current-session PATH set inline, installer handles persistent PATH). Platform detection is mandated before running either variant; PowerShell is a stated Windows requirement.
24. **End-of-run restart notice.** Step 4b documents that Claude Code snapshots hook configuration at session start; Phase 5 now closes every initial run (and any run that created or repaired hook entries) with a verbatim message instructing the user to restart the session to arm the hooks.
25. **Migration-readiness amendment: tolerant indexing + frozen artifact classes.** `phanes doc-index` gained a fallback extraction order (DOC line → first `#` heading → humanized filename), so files predating the discipline — older Phanes installs, hand-dropped files, pre-hook writes — are indexed without being edited. The 500-line ceiling was scoped to *living* documents only; frozen classes (session summaries, past-dated snapshot folders, `archive/`) are index-only — never split, retro-headered, or edited to conform, because editing history corrupts the snapshot-decay record. Conformance fixes for flagged pre-existing files are worked off lazily as T1 tasks when each file's writer next touches it; bulk conversion is forbidden. Unrecognized files inside `documentation/` are *adopted* (indexed via fallback, exempt, flagged once for user review); anything outside `documentation/` is outside Phanes' jurisdiction. Motivated by the planned PhanesUpdate.md V1→V2 migration prompt, but hardens fresh installs equally.
26. **README refreshed to v2.0.** Added Light-vs-Fleet positioning; updated pre-flight description (context7 + first-run Serena, graceful degradation, sequential-thinking removed); documented hooks, scouts, index-first navigation, universal review, Patch-Author, roster sizing, and the post-install restart notice; updated the created-files tree (`settings.json` hooks, `_index.md`, doc scripts, marker file); added the PowerShell prerequisite, a Version section pointing at this change log, and the two-entry-point table (fresh install via `phanes.md` vs. migration via `PhanesUpdate.md`, in preparation).
27. **Published to GitHub (2026-07-10).** Pushed to `Aloim/phanes` main (`ca45b18` → `88125f0`) via a temporary worktree on top of origin/main — distribution-only scope: new root `phanes.md` (v2.0), README, and this change log; the previously *published* V1 preserved verbatim at `older version/phanes.md`. Verification note: the published V1 (`ca45b18`) and the local V1 baseline (`26fdc82:PhanesLight/phanes.md`) are byte-identical after line-ending normalization — an in-session diff first flagged them as different, but the delta was CRLF/LF only. Raw install URL verified serving the v2.0 stamp. **The local monorepo main (PhanesFleet, TestProjects) remains deliberately unpublished; local main and origin/main now carry different trees by design — never plain-push or force-push local main to origin. Update the distribution repo via the same worktree-on-origin/main recipe.**

### Explicitly kept from V1 (deliberate decisions, not omissions)

* The Phanes persona, imperative register (MUST/IMPORTANT/IMPERATIVE), and motivational beats — persona conditioning demonstrably helps judgment/convention work, which is most of what sub-agents do; only pure duplication was cut.
* Parallel perspectives at **2–5 agents** for complex advisory tasks (a proposed disagreement-triggered escalation to reduce fan-out was considered and rejected).
* All verbatim payload blocks (READMEs, notice block, snapshot preamble, report template) — typo fixes only.
* R.A.C.R.S., tier system, registry two-tier design, single-writer assignments, snapshot decay discipline, `$ARGUMENTS` handling, run counter.

## Decisions taken (rationale)

| Decision | Rationale |
| --- | --- |
| Review universal, depth tiered | The safety invariant must not have exemptions; cost is controlled by scaling depth and paperwork with blast radius, not by waiving review. |
| Final Critic after Synthesizer | Audit what you apply: reviewing raw perspectives leaves the consolidated plan — the thing that ships — unaudited. |
| Indexes generated, never hand-edited | Hand-maintained indexes have as many writers as contributors — a structural Single-Writer violation. Mechanical work goes to a script; the script becomes the single writer. |
| Ceiling 500, not 300 | 300 splits eagerly and multiplies files; 500 matches the source LOC threshold, halves proliferation, and a 5–6k-token doc is still a loadable slice. Index-first navigation is what makes any ceiling safe. |
| Sonnet-first for authored code | Reviewer-as-safety-net converts Haiku's defect rate into Reflect-loop iterations; loops cost more than first-time correctness. Haiku where correctness is externally guarded. |
| Memory MCP removed | Two memory substrates (graph + doc tree) drift apart; the doc tree is auditable, diffable, and already single-writer governed. |
| Scout depth cap 1, read-only | Preserves Single-Writer structurally and bounds the token multiplication of nested spawning; the Cost Guard makes each spawn pay for itself. |
